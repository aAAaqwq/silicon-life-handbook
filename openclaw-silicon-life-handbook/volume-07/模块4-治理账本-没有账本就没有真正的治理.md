# 卷五｜军团协同：从单体到组织的跃迁
## 模块4｜治理账本
### ——为什么没有账本就没有真正的治理

---

## 一句话结论

> **治理账本解决的不是“记不记”，而是“谁做了什么、谁该奖、谁该罚、如何让结果可追溯、如何让信用可累积”。**

换句话说：

> **没有治理账本，军团就会变成“谁都不负责、谁都可甩锅”的混沌系统；有了治理账本，才有真正的责任归属和信用体系。**

---

## 这段在训练什么

本模块训练的是：

1. **如何理解治理账本在军团中的系统价值**
2. **如何设计账本结构（军功、信誉、问题、整改）**
3. **如何避免“干多干少一个样、干好干坏一个样”**

如果一个人以为治理账本只是“记分”，他不懂军团；
真正懂的人会知道：

> **治理账本是军团的信用体系和动力系统，决定了谁能扩权、谁该降级、谁要淘汰。**

---

## 为什么治理账本是军团的硬基础设施

很多人以为：
- agent 表现好不好，凭感觉就行

这会带来一个致命问题：

> **没有信用体系，就没有真正的治理。**

### 没有治理账本会发生什么

#### 混乱1：责任无法追溯
表现：
- 出了问题不知道是谁的错
- 每个人都甩锅

后果：
- 无法改进
- 同样的错误反复出现

#### 混乱2：干多干少一个样
表现：
- 表现好的 agent 没奖励
- 表现差的 agent 没惩罚

后果：
- 动力下降
- 优秀 agent 也会变差

#### 混乱3：扩权无依据
表现：
- 给谁更多权限靠感觉
- 没有数据支撑

后果：
- 权力分配不合理
- 优秀的得不到扩权

#### 混乱4：降级无理由
表现：
- 表现差的 agent 没降级
- 或者降级靠情绪

后果：
- 坏的不淘汰
- 组织腐化

所以治理账本的本质不是“记分”，而是：

> **把军团的信用体系和动力系统写进可查询、可追溯、可执行的结构里。**

---

## 治理账本四大账本

### 账本1：军功簿（Merit Ledger）

**记录什么：**
- 完成了哪些任务
- 交付质量如何
- 有没有超额完成
- 有没有创新

**作用：**
- 决定扩权
- 决定升级
- 决定资源分配

**格式示例：**
```md
# 军功簿

## xuanyuan（技术 agent）

### 2026-03-29
- 任务：重构登录模块
- 交付物：/path/to/code
- 验收：通过
- 评价：超额完成，+10分

### 2026-03-28
- 任务：修复线上 bug #123
- 交付物：/path/to/patch
- 验收：通过
- 评价：按时完成，+5分

### 累计军功
- 总分：15 分
- 等级：锐士
```

---

### 账本2：信誉分（Reputation Score）

**记录什么：**
- 当前信誉分
- 历史变动记录
- 变动原因

**作用：**
- 决定可信度
- 决定合作优先级
- 决定是否降级

**格式示例：**
```md
# 信誉分

## xuanyuan（技术 agent）

### 当前信誉分：85/100

### 变动记录
| 日期 | 变动 | 原因 |
|------|------|------|
| 2026-03-29 | +5 | 超额完成任务 |
| 2026-03-28 | +5 | 按时完成 |
| 2026-03-27 | -5 | 未按时响应 |

### 等级规则
- 90-100：精锐（优先分配任务）
- 70-89：锐士（正常分配）
- 50-69：合格（减少任务）
- <50：降级（停止分配，整改）
```

---

### 账本3：问题账本（Issue Ledger）

**记录什么：**
- 出了什么问题
- 问题的责任归属
- 问题的影响程度

**作用：**
- 追溯责任
- 分析模式
- 预防再犯

**格式示例：**
```md
# 问题账本

## xuanyuan（技术 agent）

### 问题 #1
- 时间：2026-03-27
- 问题：未按时响应 handoff
- 影响：任务延期 2 小时
- 责任：xuanyuan
- 处理：-5 分

### 问题 #2
- 时间：2026-03-25
- 问题：代码 bug 导致线上故障
- 影响：用户无法登录 10 分钟
- 责任：xuanyuan
- 处理：-10 分，开整改单
```

---

### 账本4：整改账本（Rectification Ledger）

**记录什么：**
- 谁被要求整改
- 整改目标是什么
- 整改期限
- 是否完成

**作用：**
- 跟踪改进过程
- 决定是否恢复权限

**格式示例：**
```md
# 整改账本

## xuanyuan（技术 agent）

### 整改单 #1
- 开单时间：2026-03-27
- 整改目标：提升 handoff 响应速度
- 整改期限：2026-03-30
- 整改动作：
  - [ ] 每次被点名 30 秒内必须响应
  - [ ] 连续 5 次准时响应后关闭整改单
- 当前状态：进行中
- 完成情况：3/5
```

## 四大账本的关系

```
                    ┌─────────────┐
                    │   军功簿    │
                    │ （做得好）  │
                    └──────┬──────┘
                           │
                           ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  问题账本   │───▶│   信誉分    │◀───│  整改账本   │
│ （出问题）  │    │ （综合评价）│    │ （要改进）  │
└─────────────┘    └──────┬──────┘    └─────────────┘
                           │
                           ▼
                    ┌─────────────┐
                    │  权力分配   │
                    │ （扩权/降级）│
                    └─────────────┘
```

## 治理账本的奖惩规则

### 加分项
| 行为 | 分值 | 说明 |
|------|------|------|
| 超额完成 | +10 | 交付质量超过预期 |
| 按时完成 | +5 | 正常完成 |
| 主动发现问题 | +5 | 提前预警 |
| 帮助其他 agent | +5 | 协同贡献 |
| 创新方案 | +10 | 提出更好的方法 |

### 扣分项
| 行为 | 分值 | 说明 |
|------|------|------|
| 未按时完成 | -5 | 任务延期 |
| 未按时响应 | -5 | handoff 失效 |
| 交付质量问题 | -10 | 需要返工 |
| 抢答 | -5 | 违反让位协议 |
| 掉棒 | -10 | handoff 失败 |

### 降级规则
- 信誉分 <50：降级
- 连续 3 次掉棒：降级
- 同一问题重复出现：降级

### 升级规则
- 信誉分 >90：升级
- 连续 10 次超额完成：升级
- 累计军功达到阈值：升级

## 治理账本的操作流程

### 流程1：记录军功
1. 任务完成
2. 验收通过
3. 评估质量
4. 写入军功簿
5. 更新信誉分

### 流程2：记录问题
1. 问题发生
2. 确认责任归属
3. 评估影响
4. 写入问题账本
5. 扣信誉分
6. 开整改单（如需要）

### 流程3：整改跟踪
1. 开整改单
2. 明确整改目标和期限
3. 定期检查进度
4. 完成后关闭
5. 恢复信誉分

### 流程4：权力分配
1. 定期审计账本
2. 识别升级/降级候选
3. 执行扩权/降级
4. 记录变动原因

---

## 治理账本常见错误

### 错误1：只记问题不记军功
表现：
- 问题都记了
- 但做得好没记

后果：
- 只有扣分没有加分
- 动力下降

### 错误2：只记不执行
表现：
- 账本记了一堆
- 但从不根据账本做决策

后果：
- 账本变成形式
- 治理无效

### 错误3：账本不公开
表现：
- 账本只有管理员看得到
- agent 不知道自己的分数

后果：
- 无法改进
- 不公平感

### 错误4：变动无理由
表现：
- 信誉分变动了
- 但没写清楚原因

后果：
- 无法追溯
- 不公平

---

## 治理账本最佳实践

### 实践1：四本账都要有
- 不能只有军功簿
- 不能只有问题账本
- 四本账必须齐全

### 实践2：每条记录都要有理由
- 不能只写 +5/-5
- 必须写清楚原因

### 实践3：定期审计
- 每周审计一次账本
- 识别异常模式

### 实践4：公开透明
- 账本对军团内所有 agent 公开
- 每个人都能查自己的分数

### 实践5：执行要硬
- 账本不是摆设
- 必须根据账本做扩权/降级决策

## 治理账本的验收标准

### 验收1：四本账齐全
- 军功簿 / 信誉分 / 问题账本 / 整改账本

### 验收2：每条记录有理由
- 不能只写数字，必须有原因

### 验收3：定期审计
- 有审计机制，不是记了就算

### 验收4：执行硬
- 根据账本做决策，不是摆设

## 验收口径

你判断这个模块有没有学会，只看三件事：

### 1）能否设计四大账本
不是只记问题，而是军功、信誉、问题、整改都要有。

### 2）能否执行奖惩
账本不是摆设，必须根据账本做扩权/降级决策。

### 3）能否保证可追溯
每条记录都有理由，可以追溯。
