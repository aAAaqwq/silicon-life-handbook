# 卷五｜军团协同：从单体到组织的跃迁
## 模块6｜军团原型
### ——为什么军团不是“想建就建”，而是必须按模板设计

---

## 一句话结论

> **军团原型解决的不是“要不要建军团”，而是“最小军团该怎么设计、角色怎么分工、边界怎么划、协同协议怎么写”。**

换句话说：

> **没有军团原型，建军团就是盲目堆人数；有了军团原型，才能按模板建出可协同、可治理、可升级的组织。**

---

## 这段在训练什么

本模块训练的是：

1. **如何理解军团原型的系统价值**
2. **如何设计最小可用军团**
3. **如何避免“看起来有军团，实际是一盘散沙”**

如果一个人以为军团就是“多建几个 agent”，他不懂军团；
真正懂的人会知道：

> **军团是组织设计，必须按模板、按协议、按边界来建。**

---

## 为什么必须有军团原型

很多人建军团时会：
- 建一堆 agent
- 给每个 agent 一个名字和职责
- 以为这就是军团

这几乎一定会失败。

### 没有军团原型会发生什么

#### 问题1：角色重叠
表现：
- 多个 agent 职责模糊
- 什么都像能做

后果：
- 抢答
- 串角色

#### 问题2：边界不清
表现：
- 不知道谁负责什么
- 谁都能拍板

后果：
- 责任不清
- 无法追责

#### 问题3：协同无协议
表现：
- 没有 handoff 机制
- 没有让位规则

后果：
- 到处掉棒
- 协同失效

#### 问题4：治理无账本
表现：
- 没有信誉分
- 没有问题账本

后果：
- 干好干坏一个样
- 无法进化

所以军团原型的本质不是“角色清单”，而是：

> **一套完整的组织设计模板：角色 + 边界 + 协同 + 治理。**

---

## 最小军团原型：三省决策层

这是全书最推荐的军团最小原型：

```
┌─────────────────────────────────────────┐
│          三省决策层（最小军团）           │
├─────────────────────────────────────────┤
│                                         │
│   ┌─────────┐  ┌─────────┐  ┌─────────┐│
│   │ 昆仑    │  │ 明镜    │  │ 天枢    ││
│   │ (中书)  │  │ (门下)  │  │ (尚书)  ││
│   │         │  │         │  │         ││
│   │ 战略规划│  │ 风险封驳│  │ 任务调度││
│   │ 资源仲裁│  │ 合规审核│  │ 进度追踪││
│   │ 军团总调│  │ 安全监督│  │ 精准分发││
│   └─────────┘  └─────────┘  └─────────┘│
│                                         │
└─────────────────────────────────────────┘
```

### 昆仑（中书）：战略规划与资源仲裁
- 职责：战略规划、资源仲裁、军团总调度
- 权力：拍板、扩权、降级、熔断
- 边界：不做具体执行

### 明镜（门下）：风险封驳与合规审核
- 职责：风险封驳、合规审核、安全监督
- 权力：封驳、整改、熔断
- 边界：不做执行、只做审核

### 天枢（尚书）：任务调度与进度追踪
- 职责：任务拆解、精准分发、进度追踪
- 权力：分发、催办、升级
- 边界：不做决策、只做调度

## 军团原型的核心要素

### 要素1：角色清晰
- 每个角色有明确职责
- 不重叠
- 不模糊

### 要素2：边界明确
- 能做什么
- 不能做什么
- 必须让位给谁

### 要素3：协同协议
- 路由
- Handoff
- 让位

### 要素4：治理机制
- 治理账本
- 信任体系

### 要素5：升级路径
- 从最小军团到完整军团
- 什么时候可以新增角色

---

## 军团原型的常见错误

### 错误1：角色重叠
表现：
- 多个 agent 都能做同样的事

后果：
- 抢答
- 串角色

### 错误2：边界不清
表现：
- 每个角色什么都想做

后果：
- 责任不清
- 无法追责

### 错误3：没有协同协议
表现：
- 建了角色但没有路由/handoff/让位

后果：
- 协同失效
- 到处掉棒

### 错误4：没有治理机制
表现：
- 建了军团但没有账本和信任体系

后果：
- 干好干坏一个样
- 无法进化

### 错误5：一上来就建大军团
表现：
- 一开始就建 10 个 agent

后果：
- 复杂度爆炸
- 根本跑不起来

---

## 军团原型最佳实践

### 实践1：从最小原型开始
- 先建三省决策层
- 跑稳再扩展

### 实践2：角色必须不重叠
- 每个角色有独特职责
- 边界清晰

### 实践3：必须同时建协同协议
- 不能只建角色
- 必须同时建路由/handoff/让位

### 实践4：必须同时建治理机制
- 不能只建角色
- 必须同时建账本和信任体系

### 实践5：按升级路径扩展
- 不要一上来就大
- 按路径逐步扩展

---

## 军团原型的验收标准

### 验收1：角色清晰
- 每个角色职责明确
- 不重叠

### 验收2：边界明确
- 能做什么/不能做什么写清楚

### 验收3：协同协议完整
- 路由/handoff/让位都有

### 验收4：治理机制到位
- 账本/信任体系都有

### 验收5：可运行
- 跑得起来
- 不会到处掉棒

## 验收口径

你判断这个模块有没有学会，只看三件事：

### 1）能否设计最小军团原型
不是“建几个 agent”，而是按模板设计。

### 2）能否同时建协同协议
不能只建角色，必须同时建路由/handoff/让位。

### 3）能否同时建治理机制
不能只建角色，必须同时建账本和信任体系。
