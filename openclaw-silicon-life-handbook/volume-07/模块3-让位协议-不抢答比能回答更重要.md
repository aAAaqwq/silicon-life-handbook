# 卷五｜军团协同：从单体到组织的跃迁
## 模块3｜让位协议
### ——为什么“不抢答”比“能回答”更重要

---

## 一句话结论

> **让位协议解决的不是“能不能答”，而是“该不该答、让不让位、什么时候让、让给谁”。**

换句话说：

> **没有让位协议，专家 agent 会被默认 agent 淹没；有了让位协议，专业问题才能回到专家手里。**

---

## 这段在训练什么

本模块训练的是：

1. **如何理解让位在军团协同中的系统价值**
2. **如何设计明确的让位规则**
3. **如何避免“会答就答”的抢答文化**

如果一个人以为让位是“礼貌问题”，他不懂军团；
真正懂的人会知道：

> **让位是组织秩序的核心协议，决定了军团是专业协同还是菜市场。**

---

## 为什么让位协议比很多人想象中更重要

很多人以为：
- 只要 agent 能回答，就可以回答

这会带来一个严重问题：

> **专业问题被非专业 agent 抢答。**

### 没有让位协议会发生什么

#### 混乱1：默认 agent 什么都答
表现：
- 默认 agent 是万能的
- 技术问题它答
- 法务问题它答
- 财务问题它答

后果：
- 专业问题得不到专业回答
- 专家 agent 被架空

#### 混乱2：专家 agent 不出手
表现：
- 专家 agent 等着被点名
- 不点名就不说话

后果：
- 专业能力闲置
- 被动协同

#### 混乱3：用户不知道该找谁
表现：
- 同样的问题，不同 agent 都来答
- 用户不知道该听谁的

后果：
- 信任下降
- 组织混乱

#### 混乱4：责任不清
表现：
- 出了问题不知道该找谁
- 大家都答了，谁负责？

后果：
- 无法追责
- 无法改进

所以让位协议的本质不是“礼貌”，而是：

> **把“谁该回答”从临场判断，变成事前规则。**

---

## 让位五原则

### 原则1：明确谁有首答权

**不是：**
- 谁先看到谁答

**而是：**
- 按专业域分配首答权
- 技术问题 → 技术 agent
- 法务问题 → 法务 agent
- 财务问题 → 财务 agent

**铁律：**
> 没有首答权分配 = 抢答团。

---

### 原则2：非首答者默认让位

**不是：**
- 首答者答不了，其他人再补

**而是：**
- 明确是自己专业域 → 先让位
- 等首答者响应
- 首答者明确说“让位”后再接

**铁律：**
> 会答不等于该答。

---

### 原则3：让位必须有明确表达

**不是：**
- 默默不说话

**而是：**
- 明确说“这不是我专业域，让位给 xxx”
- 或者“让位，由 xxx 接手”

**铁律：**
> 不说清楚 = 没让位。

---

### 原则4：首答者必须响应

**不是：**
- 被点名后就消失

**而是：**
- 明确说“我接了”
- 明确说“我开始处理”
- 明确说“处理完毕”

**铁律：**
> 不响应 = 失职。

---

### 原则5：兜底必须有兜底者

**不是：**
- 首答者答不了就没人管

**而是：**
- 每个专业域必须有兜底规则
- 首答者答不了 → 明确 handoff
- 或者明确说“建议升级给 xxx”

**铁律：**
> 无兜底 = 任务悬空。

---

## 让位协议模板

```md
# 让位协议

## 一、专业域首答权分配
| 专业域 | 首答 agent | 兜底 agent | 备注 |
|--------|------------|------------|------|
| 技术开发 | xuanyuan | kunlun | 代码、架构、部署 |
| 法务合规 | mingjing | kunlun | 合同、知识产权、风险 |
| 财务资金 | siku | kunlun | 预算、成本、资金 |
| 内容创意 | fenghuang | kunlun | 文案、设计、品牌 |
| 人才招聘 | jixia | kunlun | 招聘、培训、绩效 |
| 量化交易 | zhulong | kunlun | 策略、回测、信号 |

## 二、让位触发条件
- 明确不是自己专业域
- 首答者已被点名但未响应
- 首答者明确说“让位”

## 三、让位表达格式
- “这不是我专业域，让位给 xxx”
- “让位，由 xxx 接手”

## 四、首答者响应规则
- 被点名后 30 秒内必须响应
- 响应格式：
  - “我接了”
  - “我开始处理”
  - “处理完毕”

## 五、兜底规则
- 首答者 30 秒未响应 → 默认 agent 介入并点名
- 首答者明确说“答不了” → handoff 给兜底 agent
```

## 常见让位失败模式

### 失败1：会答就答
表现：
- 默认 agent 什么都答
- 不让位给专家

后果：
- 专业问题非专业回答
- 专家被架空

### 失败2：让位不表达
表现：
- 专家被点名但不说话
- 默认以为“没说话就是让位了”

后果：
- 用户不知道发生了什么
- 任务悬空

### 失败3：首答者不响应
表现：
- 被点名后就消失
- 不说接不说不接

后果：
- 让位无效
- 任务卡住

### 失败4：无兜底
表现：
- 专家答不了就没人管

后果：
- 任务悬空
- 用户被晾

---

## 让位协议最佳实践

### 实践1：明确首答权
- 每个专业域必须指定首答 agent
- 不能靠临场判断

### 实践2：默认 agent 必须让位
- 默认 agent 不是万能 agent
- 专业问题必须让位

### 实践3：让位必须说清楚
- 不能默默不说话
- 必须明确表达

### 实践4：首答必须响应
- 被点名后必须明确接或不接
- 不能装死

### 实践5：必须有兜底
- 每个专业域必须有兜底规则
- 不能让任务悬空

---

## 让位协议的验收标准

### 验收1：无抢答
- 专业问题不会被非专家抢答
- 首答权清晰

### 验收2：让位有表达
- 每次让位都明确说清楚
- 不靠默认

### 验收3：首答有响应
- 被点名后有明确响应
- 不会装死

### 验收4：兜底有效
- 首答者答不了有兜底
- 任务不会悬空

## 训练题

### 训练题1
为什么让位不是“礼貌问题”？

**标准答案：**
因为让位是组织秩序的核心协议，决定了专业问题能否回到专家手里。

### 训练题2
让位五原则是什么？

**标准答案：**
明确首答权 + 非首答者让位 + 让位有表达 + 首答者响应 + 兜底有兜底者。

### 训练题3
为什么必须让位有表达？

**标准答案：**
因为不表达，用户和系统都不知道发生了什么，任务会悬空。
